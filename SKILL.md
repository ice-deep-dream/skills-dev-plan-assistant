# 全生命周期开发助手 (Dev Plan Assistant)

> **「先想清楚，再做干净，最后记得复盘」** — 覆盖规划、开发、回顾三个阶段的完整开发闭环。

## 核心理念

这个 skill 不是帮你写代码，而是帮你**避免写出不需要的代码**。它确保你在正确的时间做正确的事：

- **开发前**：想清楚要做什么、有没有现成方案
- **开发中**：模块进度有迹可循、不遗漏不重复
- **开发后**：Bug 可追溯、经验可沉淀

## 三阶段生命周期

```
┌────────────── 阶段一：规划 ──────────────┐
│  1.1 反问理解需求                         │
│  1.2 GitHub 技术调研（避免重复造轮子）⚠️    │
│  1.3 扫描项目生成模板（从现有代码提取模式）  │
│  1.4 制定开发计划 + 生成 VuePress 站点     │
├────────────── 阶段二：开发 ──────────────┤
│  2.1 模块启动 → 创建模块开发计划           │
│  2.2 模块更新 → 更新进度状态               │
│  2.3 模块完成 → 标记 completed，更新计划   │
├────────────── 阶段三：回顾 ──────────────┤
│  3.1 Bug 文档（按需）                      │
│  3.2 测试文档（按需）                      │
│  3.3 总结文档（按需）                      │
└──────────────────────────────────────────┘
```

## 触发规则

| 阶段 | 触发条件 | 触发方式 |
|------|----------|----------|
| 阶段一 | 新对话 + 用户提出开发意图 | **自动**触发 |
| 阶段二 | 阶段一完成，用户开始编码 | **自动**触发 |
| 阶段三 | 用户明确要求创建 Bug/测试/总结 | **按需**触发 |

## 阶段一：规划（开发前）

> 详细流程见 `references/phase1-planning.md`

### 1.1 反问理解需求

在写任何代码之前，通过反问把需求搞清楚：
- 目标用户是谁？核心功能是什么？
- 技术偏好？交互形式？规模预期？
- 总结需求让用户确认：「我理解你要做的是 XXX，对吗？」

### 1.2 技术调研（避免重复造轮子）⚠️

> 详细方法论见 `references/research-methodology.md`

**跳过条件**：如果项目文件夹中已有现有项目代码（非从零开始的新项目），**直接跳过本步骤**，进入 1.3 扫描项目生成模板。

这是**最关键**的阶段。在动手写代码之前，必须确认没有现成的开源方案。

**多层次搜索策略**：
1. 提炼 3-5 组中英文搜索关键词
2. 搜索类似开源项目（awesome-selfhosted, openalternative.co, alternativeto.net）
3. 搜索工具库和框架（npm, PyPI, GitHub topics）
4. 搜索模板和脚手架

**六维评估体系**：Activity / Adoption / Security / License / Docs / Fit

**反偏见过滤**：交叉验证 3+ 独立来源，优先客观指标

### 1.3 扫描项目生成模板

> 详细流程见 `references/phase1-template.md`

在写开发计划之前，先扫描项目现有代码，提取已有的模式做成模板文档。**这不是代码规范，而是从实际代码中提取的「项目已经怎么做的」快照**。

**扫描内容**：
- 前端：全局 API 调用方式、store 结构、通用组件、路由模式
- 后端：全局配置、中间件、错误处理、数据库操作模式
- 全栈：目录结构、文件命名规律、模块组织方式

**输出**：`docs/03-模板中心/`，前端/后端各一个完整模板文档，包含技术栈、目录结构、代码骨架、编码规范和开发 Checklist。

### 1.4 制定开发计划 + 生成站点

一个模块一个计划文档，写入 `docs/04-开发计划/`。完成模板扫描后将 `assets/vuepress-template/` 复制到项目 `docs/`，运行 `npm install && npm run docs:dev`。

## 阶段二：开发（开发中）

> 详细流程见 `references/phase2-development.md`

### 自动行为

| 场景 | 行为 |
|------|------|
| 用户提出新模块需求 | 创建模块开发计划，写入 `docs/04-开发计划/`，参照 `03-模板中心/` |
| 开发过程中 | 按需更新模块进度 |
| 用户确认模块完成 | 标记模块为 completed，进度 100%，**更新开发计划** |

### 一个模块一个计划

**模块定义**：这里的「模块」是**架构师视角的功能域**，不是代码层面的小模块。一个模块包含前后端全链路，覆盖完整业务流程。

**示例 — 用户认证模块**：
- 前端：登录页、注册页、验证码组件、Token 存储、路由守卫
- 后端：登录接口、注册接口、验证码生成/校验、JWT 签发、密码加密
- 数据库：用户表、验证码表、登录日志
- 全流程：注册 → 验证 → 登录 → Token 刷新 → 登出

**不拆分**成「前端登录模块」「后端登录模块」，而是作为一个完整的**功能域**，一个文档覆盖前后端全链路。

每个模块独立一个文档，使用模板中心中提取的模式作为参照。模块完成后必须更新计划文档的状态和进度。

### 默认不主动写文档

开发过程中**不主动创建或修改文档**，除非：
- 模块启动/完成（自动行为）
- 用户明确要求记录
- AI 判断关键信息需要保存（反问确认后写入）

## 工作量校准

| 任务规模 | Phase 1.2 调研深度 | 跳过什么 |
|----------|-------------------|----------|
| 已有项目（非从零开始） | **跳过** | 全部跳过，直接进入 1.3 |
| 简单任务（修 bug、小功能） | 1 分钟快速搜索 | 多层评估 |
| 中等任务（新模块） | 3-5 分钟 | 替代品数据库 |
| 大型任务（新产品、重构） | 10-15 分钟 | 无，完整五步 |

## 跨平台兼容性

此 skill 设计为跨 AI 平台兼容（Trae、WorkBuddy、Claude Code 等）：

- 不依赖平台特定功能，所有操作通过通用命令完成
- 标准化 Markdown + YAML frontmatter
- 各平台自动适配对应工具

## 快速参考

### 用户说「跳过 XX」时
- 跳过阶段一（需求理解）→ 提醒风险，但还是听用户的
- 跳过技术调研 → 已有项目自动跳过；新项目**强烈建议不要跳过**，但尊重用户决定
- 跳过模板扫描 → 对于新项目可以跳过（没有代码可扫），老项目建议先跑一次
- 跳过生成站点 → 确认是否有其他文档管理方式

### 最小化运行
如果用户只想快速开始，至少完成：
1. 一段话总结需求
2. 已有项目跳过 GitHub 搜索；新项目至少一次搜索（确认不是重复造轮子）
3. 扫描项目模板（如果有现有代码）
4. 一个简单的模块列表
