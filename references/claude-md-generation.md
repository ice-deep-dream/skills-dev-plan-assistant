# CLAUDE.md 生成规范

> 在阶段一规划时自动生成，只执行一次。如果已存在则补充，不覆盖原有内容。

## 执行时机

| 项目状态 | 行为 |
|----------|------|
| 全新项目 | 在 1.4 制定开发计划后生成 |
| 已有项目 | 在 1.3 扫描模板后补充（追加未覆盖的内容） |

## 核心理念

**CLAUDE.md 是给 AI 看的「项目说明书」**，帮助 AI 理解：
- 这个项目是做什么的
- 用了什么技术栈
- 代码怎么组织的
- 有什么特殊约定

用户可能不懂技术、描述模糊、用词不专业，AI 需要**以产品经理思维理解需求**，翻译成技术语言写入 CLAUDE.md。

## 生成步骤

### Step 1：收集信息

从以下来源提取信息：

| 信息 | 来源 |
|------|------|
| 项目名称 | package.json / 用户描述 |
| 技术栈 | package.json / pom.xml / 目录结构 |
| 目录结构 | 遍历项目根目录 |
| 核心功能 | 用户需求描述 / 现有代码分析 |

### Step 2：理解用户需求（产品经理思维）

用户描述可能：
- 用词不专业：「我要一个能看数据的页面」→ 数据展示/仪表盘
- 模糊：「要快」「要好用」→ 性能要求/用户体验
- 混乱：东一句西一句 → 需要整合归纳

**处理方式**：
```
用户说：「我要一个那个...就是能管用户的，然后能看到他们干了啥」
    ↓
产品经理理解：
  - 用户管理：增删改查用户
  - 操作日志：记录并展示用户行为
    ↓
写入 CLAUDE.md：
  ## 核心功能
  | 功能 | 说明 |
  |------|------|
  | 用户管理 | 管理员可增删改查用户 |
  | 操作日志 | 记录用户操作，支持查询展示 |
```

### Step 3：生成/补充文档

**全新项目**：
```
1. 创建 CLAUDE.md
2. 写入模板结构
3. 填充已知信息
4. 标记未知部分，等待后续补充
```

**已有项目（CLAUDE.md 已存在）**：
```
1. 读取现有 CLAUDE.md
2. 对比模板结构
3. 追加缺失章节（不覆盖已有内容）
4. 更新过时信息（需用户确认）
```

## 用户需求翻译表

常见的用户描述 → 技术翻译：

| 用户描述 | 技术理解 |
|----------|----------|
| 「要快」「不能卡」 | 性能要求：响应时间 < 200ms / 支持并发 |
| 「要安全」「不能被黑」 | 安全要求：认证授权、数据加密、防注入 |
| 「要好维护」「以后能改」 | 代码规范：模块化、文档齐全、测试覆盖 |
| 「要好看」「漂亮点」 | UI/UX：使用组件库、响应式设计 |
| 「要能用手机看」 | 移动端适配：响应式 / 移动端优先 |
| 「数据不能丢」 | 数据持久化：数据库备份、事务支持 |
| 「能导出」 | 导出功能：Excel/PDF 导出 |
| 「能搜索」 | 搜索功能：全文检索 / 过滤筛选 |
| 「要登录才能用」 | 权限控制：用户认证、角色授权 |
| 「要记录谁干了啥」 | 审计日志：操作记录、行为追踪 |

## 执行标记

为避免重复生成，在项目根目录创建 `.claude-md-generated` 标记文件：

```
# .claude-md-generated 内容示例
generated_at: 2026-06-18
version: 1
```

每次检查：如果文件存在，跳过生成步骤。

## 注意事项

1. **只执行一次** — 通过标记文件判断
2. **不覆盖已有内容** — 使用追加模式
3. **用产品语言** — 功能描述要让用户能看懂
4. **保持更新** — 如果用户说「改一下」，可以更新对应章节
5. **简洁实用** — 不要写成说明书，只写必要信息
