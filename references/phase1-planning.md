# 阶段一：规划 — 详细工作流

## 1.1 反问理解需求

### 目标
在写任何代码之前，通过反问把需求搞清楚。

### 执行步骤

**第一轮：核心信息**
- 目标用户是谁？谁会用这个东西？
- 核心功能是什么？最核心的 1-3 个功能
- 技术偏好？有指定的技术栈吗？

**第二轮：补充信息**
- 交互形式？Web 应用？CLI 工具？API 服务？
- 规模预期？个人工具？团队协作？面向公众？
- 有什么现成参考？有没有类似产品可以参考？

**第三轮：确认总结**
将需求总结为一段话，让用户确认：「我理解你要做的是 XXX，对吗？」

### 输出到文档
将确认后的需求写入 `docs/01-需求分析/README.md`

### 注意事项
- 不要一次问太多问题（每次 2-3 个），分轮次逐步推进
- 如果用户说「随便」「你看着办」，给出合理的默认建议并让用户确认
- 需求确认后，告诉用户：「需求已确认，接下来我会检索 GitHub 看有没有现成的方案」

## 1.2 技术调研（避免重复造轮子）

> 详细方法论见 `references/research-methodology.md`

**跳过条件**：如果项目文件夹中已有现有项目代码（非从零开始的新项目），**直接跳过本步骤**，进入 1.3 扫描项目生成模板。

### 核心原则
**在动手写代码之前，必须确认没有现成的开源方案**。花 2 分钟搜索可能省 2 周的开发时间。

### 快速流程
1. 提炼 3-5 组搜索关键词（中英文）
2. 搜索类似开源项目
3. 搜索工具库和框架
4. 评估是否可用
5. 记录结论

### 输出到文档
将所有发现写入 `docs/02-技术调研/README.md`，必须包含：
- 搜索关键词列表
- 候选项目对比表格
- **明确结论**：用还是不用？参考还是从零做？

---

## 1.3 扫描项目生成模板

> 详细流程见 `references/phase1-template.md`

技术调研完成后，如果项目已有代码（老项目加新功能），必须先扫描现有代码提取模式。

### 核心理念
这不是定义代码规范，而是从已有代码中提取「项目已经怎么做的」，做成模板，让新模块有参照。

### 快速流程
1. 判断项目类型（前端/后端/全栈）
2. 遍历关键目录（api/、store/、router/、components/ 等）
3. 找 2-3 个典型文件，提取核心结构（去业务逻辑，留骨架）
4. 写入 `docs/03-模板中心/`

### 全新项目
如果没有代码可扫，跳过这一步。

---

## 1.4 制定开发计划 + 生成站点

### 执行步骤

1. **一个模块一个计划**：每个模块独立一个文档，参照模板中心
2. **复制模板**：将 `assets/vuepress-template/` 复制到项目的 `docs/` 目录
3. **填充内容**：将 1.1、1.2、1.3 的内容填入对应文档
4. **编写开发计划**：在 `docs/04-开发计划/` 中为每个模块创建计划文档
5. **初始化站点**：运行 `cd docs && npm install`
6. **启动预览**：运行 `npm run docs:dev`

### 开发计划文档必须包含
- Frontmatter：status、progress、priority、owner、created、updated
- 计划要具体到模块级别
- 为每个模块标注优先级和依赖关系
- 模块完成后必须更新状态和进度
